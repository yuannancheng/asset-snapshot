# 产品设计书

> 产品名：资产快照

# 🧾 一、产品定位（定死，防跑偏）

👉 **个人资产快照与趋势分析工具（本地优先）**

不是记账软件，不做流水。

核心能力：

- 记录某一时间点的资产分布
- 自动计算总资产 / 可用资产
- 展示资产趋势 & 分布

------

# 🧱 二、核心设计原则（必须遵守）

### 1️⃣ 只记录“结果”，不记录“过程”

- ✔ 记录资产快照
- ❌ 不记录每笔消费

------

### 2️⃣ 所有“汇总数据”不入库

```text
总资产 / 可用资产 = 实时计算
```

------

### 3️⃣ 分类可配置，但计算规则固定

- 分类：用户自由定义
- 计算：系统控制（资产属性）

------

### 4️⃣ 本地优先 + 单文件

- 一个数据文件（类似 `.db`）
- 可选加密

------

# 🗂️ 三、数据模型（最终定稿 ✅）

这是最关键部分，我帮你压到**刚好够用 + 不会后悔**。

------

## 1️⃣ 一级分类表（platforms）

👉 表示“钱在哪个平台”

```text
platforms
---------
id (PK)
name            // 支付宝 / 招行 / 微信
sort_order      // 排序
created_at
```

------

## 2️⃣ 二级分类表（accounts）

👉 实际录入金额的单位

```text
accounts
--------
id (PK)
platform_id (FK -> platforms.id)

name            // 余额 / 理财 / 花呗
type            // asset_liquid / asset_nonliquid / debt

sort_order
is_active       // 是否启用
created_at
```

------

## 💡 type 枚举（关键设计）

```text
asset_liquid        // 流动资产（计入可用资产）
asset_nonliquid     // 非流动资产（计入总资产）
debt                // 负债（扣减）
```

👉 不允许用户自定义（否则会乱）

------

## 3️⃣ 快照表（snapshots）

👉 一次记录（通常一个月一次）

```text
snapshots
---------
id (PK)
date            // 记录日期（核心）
note            // 备注（可选）
created_at
```

------

## 4️⃣ 快照明细表（snapshot_items）

👉 每个账户在该时间点的金额

```text
snapshot_items
--------------
id (PK)
snapshot_id (FK -> snapshots.id)
account_id  (FK -> accounts.id)

amount      // 金额（支持负数，但一般不用）

created_at
```

------

## 🧠 数据关系（一句话理解）

```text
一个 snapshot
  → 对应多个 snapshot_items
    → 每个 item 对应一个 account
      → account 属于一个 platform
```

------

# 🧮 四、核心计算逻辑（必须统一）

------

## 1️⃣ 总资产

```text
总资产 =
  所有 asset_liquid +
  所有 asset_nonliquid -
  所有 debt
```

------

## 2️⃣ 可用资产

```text
可用资产 =
  所有 asset_liquid
```

------

## 3️⃣ 平台资产（用于图表）

```text
某平台资产 =
  该平台下所有账户汇总（含负债）
```

------

# 📊 五、图表系统设计

------

## 1️⃣ 图表类型（固定3个）

### ✔ 总资产趋势（主图）

- X：时间
- Y：总资产

------

### ✔ 平台趋势图

- 每个平台一条线
- 展示资金迁移

------

### ✔ 资产分布图（饼图）

- 某一时间点
- 各平台占比

------

## 2️⃣ 时间筛选（必须有）

```text
- 最近30天
- 最近3个月
- 最近1年
- 自定义区间
```

------

## 3️⃣ 自动聚合（防止图太密）

| 时间范围 | 数据粒度 |
| -------- | -------- |
| ≤1个月   | 按天     |
| ≤1年     | 按周     |
| >1年     | 按月     |

------

# 🔐 六、安全设计（最终版）

------

## 1️⃣ 模式

| 模式     | 说明          |
| -------- | ------------- |
| 无密码   | 不加密        |
| 加密模式 | AES加密数据库 |

------

## 2️⃣ 加密方案

- SQLite + SQLCipher
- AES-256

------

## 3️⃣ 密钥派生

- Argon2（推荐）
- 或 PBKDF2

------

## 4️⃣ 防暴力破解

### 延迟策略：

```text
1-3次：无延迟
4次：2秒
5次：5秒
6次：10秒
7次：30秒
8次+：60秒（封顶）
```

------

## 5️⃣ 限制

- ❌ 不支持找回密码
- ✔ 支持修改密码

------

## 🟣 查看数据

- 看趋势图

------

❌ 不做消费分类
❌ 不做预算系统

# ⚠️ 九、几个关键技术约束（很重要）

------

## 1️⃣ 不存这些字段

```text
❌ 总资产
❌ 可用资产
```

------

## 2️⃣ 必须支持

- 分类增删改

# 🧩十、技术实现方案（UI & 跨平台补充）

------

## 🧭 1️⃣ 技术选型（最终确定）

👉 前端 UI 与桌面框架采用：

- 桌面框架：Tauri
- 前端框架：React（或 Vue 二选一，推荐 React）
- 样式系统：Tailwind CSS
- UI 组件：Headless UI

------

### 🎯 选型目标

满足以下要求：

- ✅ 多平台（Windows / macOS / Linux）
- ✅ 安装包体积小（显著小于 Electron）
- ✅ UI 现代化（类 Web App / SaaS 风格）
- ✅ 开发效率高（组件生态成熟）

------

### ⚖️ 技术决策说明

#### 1️⃣ 不使用 Electron

原因：

- 安装体积过大（通常 >100MB）
- 内存占用高
- 不符合“轻量工具”定位

------

#### 2️⃣ 不采用原生 GUI（egui / Slint）

原因：

- UI 风格偏工具化
- 动效能力有限
- 难以实现现代 Web 风格界面

------

#### 3️⃣ 选择 Tauri 的核心原因

- 使用系统 WebView（不内置浏览器）
- 安装包体积小（通常 5–15MB）
- 性能优于 Electron
- 可结合 Rust 实现本地能力（加密 / SQLite）

------

## 🧱 2️⃣ 总体架构设计

------

### 架构分层

```text
前端（Web UI）
  ↓
Tauri Bridge（API 调用）
  ↓
Rust 后端
  ↓
SQLite / SQLCipher
```

------

### 各层职责

#### 🟢 前端（React + Tailwind）

负责：

- 页面 UI 渲染
- 图表展示
- 用户交互
- 表单输入

------

#### 🟣 Tauri Bridge

负责：

- 前端 ↔ Rust 通信
- 调用本地能力（文件 / 数据库 / 加密）

------

#### 🔵 Rust 后端

负责：

- 数据读写（SQLite）
- 加密逻辑（SQLCipher / 密钥处理）
- 核心计算（资产汇总）

------

#### ⚫ 数据层

- SQLite（单文件）
- 可选 SQLCipher 加密

------

## 🧩 3️⃣ UI 设计规范（必须遵守）

------

### 🎨 整体风格

👉 统一为：

**现代 Web App 风格（SaaS 风格）**

特点：

- 大量留白
- 扁平化设计
- 柔和阴影
- 圆角组件
- 清晰层级

------

### 🧱 组件策略

#### 使用 Headless UI 的原因：

- 只提供“行为”，不限制样式
- 可完全自定义视觉风格
- 与 Tailwind 高度兼容

------

#### 📦 推荐组件分层

| 层级     | 内容                   |
| -------- | ---------------------- |
| 基础组件 | Button / Input / Modal |
| 组合组件 | 表单 / 卡片 / 图表容器 |
| 页面组件 | Dashboard / 录入页     |

------

### 🎯 关键 UI 要求

必须满足：

- ✅ 所有交互有 hover / active 状态
- ✅ 弹窗带过渡动画
- ✅ 表单有明确反馈（成功 / 错误）
- ✅ 支持暗色模式（建议但不强制）

------

## 📊 4️⃣ 图表实现建议

------

### 推荐方案：

- Recharts（React）
  或
- ECharts（更强但稍重）

------

### 图表要求

- 风格统一（颜色 / 字体）
- 支持 tooltip
- 支持时间范围切换
- 动画不过度（避免性能浪费）

------

## 🔐 5️⃣ 前后端职责边界（重要）

------

### ❌ 前端不负责：

- 数据加密
- 密码处理
- 核心计算逻辑

------

### ✅ Rust 必须负责：

- 密码校验
- 数据加解密
- 资产计算（总资产 / 可用资产）

------

👉 原则：

> **所有“可信逻辑”必须在 Rust 层**

------

## ⚡ 6️⃣ 性能与体积控制

------

### 🎯 目标

- 安装包：≤ 20MB
- 启动时间：< 1秒
- 内存占用：低于 Electron 应用

------

### 优化策略

#### 前端：

- Tailwind 按需构建（Purge）
- 避免大型 UI 库（如 antd 全量引入）
- 图表按需加载

------

#### Tauri：

- 关闭未使用 API
- 精简权限配置

------

#### Rust：

- Release 构建（开启优化）
- 控制依赖数量

------

## 🧪 7️⃣ 开发阶段建议

------

### 阶段 1（快速可用）

- UI 用基础组件即可
- 优先实现核心流程（snapshot 录入 + 展示）

------

### 阶段 2（体验优化）

- 加入动画
- 优化布局
- 调整配色

------

### 阶段 3（完善）

- 暗色模式
- 交互细节优化
- 性能优化

------
