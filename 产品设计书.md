# 产品设计书

> 产品名：资产快照

# 🧾 一、产品定位（定死，防跑偏）

👉 **个人资产快照与趋势分析工具（本地优先）**

不是记账软件，不做流水。

核心能力：

- 记录某一时间点的资产分布
- 自动计算总资产 / 可用资产
- 展示资产趋势 & 分布

------

# 🧱 二、核心设计原则（必须遵守）

### 1️⃣ 只记录“结果”，不记录“过程”

- ✔ 记录资产快照
- ❌ 不记录每笔消费

------

### 2️⃣ 所有“汇总数据”不入库

```text
总资产 / 可用资产 = 实时计算
```

------

### 3️⃣ 分类可配置，但计算规则固定

- 分类：用户自由定义
- 计算：系统控制（资产属性）

------

### 4️⃣ 本地优先 + 单文件

- 一个数据文件（类似 `.db`）
- 可选加密

------

# 🗂️ 三、数据模型（最终定稿 ✅）

这是最关键部分，我帮你压到**刚好够用 + 不会后悔**。

------

## 1️⃣ 一级分类表（platforms）

👉 表示“钱在哪个平台”

```text
platforms
---------
id (PK)
name            // 支付宝 / 招行 / 微信
sort_order      // 排序
created_at
```

------

## 2️⃣ 二级分类表（accounts）

👉 实际录入金额的单位

```text
accounts
--------
id (PK)
platform_id (FK -> platforms.id)

name            // 余额 / 理财 / 花呗
type            // asset_liquid / asset_nonliquid / debt

sort_order
is_active       // 是否启用
created_at
```

------

## 💡 type 枚举（关键设计）

```text
asset_liquid        // 流动资产（计入可用资产）
asset_nonliquid     // 非流动资产（计入总资产）
debt                // 负债（扣减）
```

👉 不允许用户自定义（否则会乱）

------

## 3️⃣ 快照表（snapshots）

👉 一次记录（通常一个月一次）

```text
snapshots
---------
id (PK)
date            // 记录日期（核心）
note            // 备注（可选）
created_at
```

------

## 4️⃣ 快照明细表（snapshot_items）

👉 每个账户在该时间点的金额

```text
snapshot_items
--------------
id (PK)
snapshot_id (FK -> snapshots.id)
account_id  (FK -> accounts.id)

amount      // 金额（支持负数，但一般不用）

created_at
```

------

## 🧠 数据关系（一句话理解）

```text
一个 snapshot
  → 对应多个 snapshot_items
    → 每个 item 对应一个 account
      → account 属于一个 platform
```

------

# 🧮 四、核心计算逻辑（必须统一）

------

## 1️⃣ 总资产

```text
总资产 =
  所有 asset_liquid +
  所有 asset_nonliquid -
  所有 debt
```

------

## 2️⃣ 可用资产

```text
可用资产 =
  所有 asset_liquid
```

------

## 3️⃣ 平台资产（用于图表）

```text
某平台资产 =
  该平台下所有账户汇总（含负债）
```

------

# 🧩十、技术实现方案（UI & 跨平台补充）

------

## 🧭 1️⃣ 技术选型（最终确定）

👉 前端 UI 与桌面框架采用：

- 桌面框架：Tauri
- 前端框架：React（或 Vue 二选一，推荐 React）
- 样式系统：Tailwind CSS
- UI 组件：Headless UI

------

### 🎯 选型目标

满足以下要求：

- ✅ 多平台（Windows / macOS / Linux）
- ✅ 安装包体积小（显著小于 Electron）
- ✅ UI 现代化（类 Web App / SaaS 风格）
- ✅ 开发效率高（组件生态成熟）

------

### ⚖️ 技术决策说明

#### 1️⃣ 不使用 Electron

原因：

- 安装体积过大（通常 >100MB）
- 内存占用高
- 不符合“轻量工具”定位

------

#### 2️⃣ 不采用原生 GUI（egui / Slint）

原因：

- UI 风格偏工具化
- 动效能力有限
- 难以实现现代 Web 风格界面

------

#### 3️⃣ 选择 Tauri 的核心原因

- 使用系统 WebView（不内置浏览器）
- 安装包体积小（通常 5–15MB）
- 性能优于 Electron
- 可结合 Rust 实现本地能力（加密 / SQLite）

------

## 🧱 2️⃣ 总体架构设计

------

### 架构分层

```text
前端（Web UI）
  ↓
Tauri Bridge（API 调用）
  ↓
Rust 后端
  ↓
SQLite / SQLCipher
```

------

### 各层职责

#### 🟢 前端（React + Tailwind）

负责：

- 页面 UI 渲染
- 图表展示
- 用户交互
- 表单输入

------

#### 🟣 Tauri Bridge

负责：

- 前端 ↔ Rust 通信
- 调用本地能力（文件 / 数据库 / 加密）

------

#### 🔵 Rust 后端

负责：

- 数据读写（SQLite）
- 加密逻辑（SQLCipher / 密钥处理）
- 核心计算（资产汇总）

------

#### ⚫ 数据层

- SQLite（单文件）
- 可选 SQLCipher 加密

------

## 🧩 3️⃣ UI 设计规范（必须遵守）

------

### 🎨 整体风格

👉 统一为：

**现代 Web App 风格（SaaS 风格）**

特点：

| 基础组件 | Button / Input / Modal |

### 🎯 关键 UI 要求

必须满足：

- ✅ 所有交互有 hover / active 状态
- ✅ 弹窗带过渡动画

## 📊 4️⃣ 图表实现建议

------

### 推荐方案：

  或
- ECharts（更强但稍重）

------

### 图表要求

- 风格统一（颜色 / 字体）

- 动画不过度（避免性能浪费）

------

## 🔐 5️⃣ 前后端职责边界（重要）

------

### ❌ 前端不负责：

- 数据加密

👉 原则：

## ⚡ 6️⃣ 性能与体积控制

### 🎯 目标

# 📝 十一、快照变动分析系统（Snapshot Change Analysis）

------

## 🧭 功能定位

用于记录和解释两次资产快照之间的资产变化原因。

本功能不是记账系统，不记录全部消费流水，仅记录对资产变化具有解释价值的重要收入与支出项目，并自动生成人类可读的资产变动说明。

典型用途：

```text
为什么本月总资产增加了？

为什么本月总资产减少了？

本期资产变化主要来自哪些事项？
```

------

## 🎯 设计目标

帮助用户在记录资产快照后，快速完成资产变化归因分析。

实现：

- 自动计算两次快照的资产变化
- 记录主要收入项目
- 记录主要支出项目
- 自动计算未解释差额
- 自动生成变动说明
- 长期保存分析记录

------

## 📍 功能入口

在快照详情页增加入口：

```text
[资产变动分析]
```

或：

```text
[生成变动说明]
```

点击后进入分析界面。

------

## 📊 分析界面

系统自动展示：

```text
本期总资产
205000

上期总资产
204300

资产变化
+700
```

计算规则：

```text
资产变化 =
本期总资产 - 上期总资产
```

说明：

```text
正数 = 总资产增加

负数 = 总资产减少
```

------

## 💰 收入项目管理

支持添加多个收入项目。

每个项目包含：

```text
名称
金额记录列表
自动汇总金额
```

示例：

```text
项目奖金收入

5000

合计
5000
```

------

## 💸 支出项目管理

支持添加多个支出项目。

每个项目包含：

```text
名称
金额记录列表
自动汇总金额
```

示例：

```text
交通支出

120
80
60
60

合计
320
```

------

## 🧾 多金额记录支持

每个收入或支出项目允许录入多个金额。

示例：

```text
餐饮支出

80
120
90
160
```

系统自动计算：

```text
合计
450
```

设计目的：

- 保留用户原有 Excel 使用习惯
- 无需手工计算汇总金额
- 支持快速归类多个相关支出

------

## 🧮 自动归因计算

系统实时计算：

```text
已解释金额 =
收入项目总额
-
支出项目总额
```

示例：

```text
项目奖金收入
+5000

房租支出
-1800

交通支出
-320

餐饮支出
-450

家电支出
-1500
```

结果：

```text
已解释金额

+930
```

------

## 🔍 未解释金额

系统自动计算：

```text
未解释金额 =
资产变化 - 已解释金额
```

示例：

```text
实际资产变化
+700

已解释金额
+930

未解释金额
-230
```

界面显示：

```text
未解释金额

-230
```

------

## ⚡ 自动补足差额

提供按钮：

```text
[自动补足差额]
```

------

### 情况一：未解释金额小于零

```text
未解释金额

-230
```

自动创建：

```text
其余支出

230
```

------

### 情况二：未解释金额大于零

```text
未解释金额

+500
```

自动创建：

```text
其余收入

500
```

------

补足后：

```text
未解释金额

0
```

显示：

```text
✓ 已完全解释
```

------

## ✍️ 变动说明生成

金额输入框失去焦点后：

自动生成资产变动说明文本。

------

### 生成规则

收入项目：

```text
名称 + 金额
```

例如：

```text
项目奖金收入5000元
```

------

支出项目：

```text
名称 + 金额
```

例如：

```text
房租支出1800元
```

------

最后追加总结：

```text
总资产增加XXX元
```

或：

```text
总资产减少XXX元
```

------

### 生成示例

```text
项目奖金收入5000元，
房租支出1800元，
家电支出1500元，
餐饮支出450元，
交通支出320元，
其余支出230元，
总资产增加700元。
```

------

### 📋 排序规则

生成说明时按照金额绝对值降序排列：

```text
项目奖金收入5000元，
房租支出1800元，
家电支出1500元，
餐饮支出450元，
交通支出320元，
其余支出230元，
总资产增加700元。
```

目的：

优先展示影响最大的变化原因。
